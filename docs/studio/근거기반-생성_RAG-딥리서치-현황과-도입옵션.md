# 문서공방 근거 기반 생성 — 현황과 도입 옵션 (RAG·딥리서치)

> 목적: KOICA 공식문서 초안의 **환각(hallucination) 위험**을 줄이기 위한 현재 구조와,
> Odysseus가 이미 보유해 **가져와 연결할 수 있는** 기능을 정리한다.
> (배경: 'AI 환각'으로 컨설팅 보고서가 철회된 사례에 대한 경계.)

작성일: 2026-06-19 · 대상 코드: `services/studio/*`

---

## 1. 현재 문서공방은 어떻게 초안을 만드나 (사실)

문서공방(스튜디오)의 생성 경로는 **단순 1회 LLM 호출**이다. RAG도, 딥리서치도 **쓰지 않는다.**

```
업로드 입력파일 → 텍스트 추출(extractors.py) → [추출 텍스트 + 지시 + 레시피 프롬프트]
   → LLM 1회 호출(agy/codex)  → 결과를 hwpx/pptx/xlsx/docx + md 로 렌더
```

- 근거의 출처는 **사용자가 첨부한 입력 파일**뿐이다("input-grounded"). 외부 자료를 스스로 찾지 않는다.
- 코드 위치: 호출부 `services/studio/pipeline.py` (`_build_prompt`, `_generate_content`),
  LLM 래퍼 `services/studio/llm.py` (`generate_text`/`generate_json` → `chat`).
- **검증 결과**: `services/studio` 전체에 벡터검색/임베딩/딥리서치/웹검색 호출이 없음(코드 grep 확인).
  Odysseus 메뉴의 "Deep Research"·"Brain"은 별개 기능이며 문서공방 파이프라인과 **연결돼 있지 않다.**

### 현재의 환각 방어 수단 (프롬프트·사람 검토 수준)
- 레시피 프롬프트에 명시: "사실은 입력 자료에 근거하고, 불확실하면 임의로 지어내지 말고 **[확인 필요]**로 표시".
- 수치·일정·고유명사는 입력 자료 표기를 따르도록 지시.
- 생성 직후 **.md 미리보기**를 사람이 검토한 뒤 다운로드(최종 책임은 사람).

> 요약: 현재는 "첨부 자료 + 프롬프트 규율 + 사람 검토"로 환각을 억제한다.
> 자동 사실검증이나 외부 근거 보강은 **아직 없다.**

## 2. Odysseus가 이미 가진 기능 (외부에서 찾을 필요 없음)

아래는 **이 저장소에 이미 구현돼 있는** 기능이다. 문서공방에 *연결*하는 일이 남았을 뿐, 새로 찾을 필요는 없다.

| 기능 | 핵심 파일 | 하는 일 | 외부 의존성 |
|---|---|---|---|
| **RAG(벡터 검색)** | `src/rag_vector.py`, `src/rag_singleton.py`, `src/embeddings.py` | 문서를 임베딩→유사 문단 검색(하이브리드: 벡터+키워드). 조직 자료(KOICA 가이드·정책·기존 산출물)를 색인해두면 생성 시 자동 참고 | **ChromaDB 서버** + **임베딩 모델**(Ollama 또는 로컬 fastembed). agy/codex만으로는 안 됨 |
| **딥리서치** | `src/deep_research.py`, `services/research/service.py` | 질문→웹 다중 라운드 검색→소스별 추출→종합 보고서(+인라인 출처). 최대 8라운드/수분 소요 | **웹 접근** + 검색 공급자(아래) |
| **웹 검색** | `services/search/{core,providers,content}.py` | 검색 질의→결과/본문 추출. 공급자: SearXNG·DuckDuckGo(무료), Brave·Google PSE·Tavily·Serper(API키) | 온라인 + (공급자에 따라) API키 |
| **출처/인용** | `services/research/service.py`, `src/url_safety.py` | 결과에서 `[제목](URL)` 출처 추출·정리, SSRF/사설IP 차단 | 없음 |
| **목표기반 추출** | `src/goal_based_extractor.py` | "목표"에 맞춰 긴 텍스트에서 근거 문단만 골라 요약(JSON: 근거/인용/요약) | LLM |

> 위 항목·경로는 코드 탐색으로 확인했다. 다만 **문서공방 연결 코드는 아직 작성하지 않았다**(설계/구현 과제).

## 3. 도입 옵션 (난이도·효과·전제)

문서공방에 붙일 때의 현실적 순서. 효과 수치는 검증 전이므로 단정하지 않는다.

| 옵션 | 난이도 | 전제(중요) | 적합도 |
|---|---|---|---|
| **A. 프롬프트 강화** — "주장마다 근거/출처 명시, 불확실은 [확인 필요]" 강제 | 낮음 | 없음(즉시) | 항상 |
| **B. RAG 보강** — KOICA 기본자료를 색인 후, 생성 시 관련 문단 자동 주입 | 중간 | ChromaDB + 임베딩 모델 구동, 사전 색인 | **공식문서에 가장 적합**(일관성·기관자료 근거) |
| **C. 웹 검색 보강** — 레시피에 "온라인 자료 포함" 옵션, 최신 정책·통계 주입 | 낮음~중간 | 온라인 + 검색 공급자 | 최신성 필요 시 |
| **D. 딥리서치** — "상세 조사" 선택 시 다중 라운드 후 종합 | 높음 | 비동기 처리 재설계 + 웹 + 수분 소요 | 심층 배경조사용 |

### 폐쇄망/오프라인 주의
- RAG(B)는 **인터넷 없이도** 가능하지만 ChromaDB 서버와 임베딩 모델(로컬 fastembed 권장)이 필요하다.
- 웹검색(C)·딥리서치(D)는 **인터넷이 필수**다. KOICA 보안망에서 쓸 수 없을 수 있으니 사전 확인 필요.

## 4. 권고 (환각 최소화 관점)

1. **즉시**: 옵션 A(프롬프트 규율)로 출처 명시·[확인 필요] 강제 — 이미 부분 적용 중, 더 강화 가능.
2. **다음**: 옵션 B(RAG)로 KOICA 기본자료(가이드·정책·과거 산출물)를 색인해 **기관 근거**를 자동 첨부.
   공식문서 특성상 웹검색보다 RAG가 더 안전·일관적이다.
3. **사람 검토는 항상 유지**: 자동화가 늘어도 최종 사실확인은 사람이 한다(컨설팅사 사고의 교훈).

> 어떤 옵션을 도입할지는 별도 결정 사항이다. 본 문서는 "가져올 수 있는 것"의 목록과 전제를 정리한 것이며,
> 실제 연결 코드는 아직 들어가지 않았다.

### 의사결정 메모 (2026-06-19)
- **옵션 A(프롬프트 강화) = 진행함.** `services/studio/pipeline.py`의 `_build_prompt`에 전역
  "정확성·환각 방지 규칙"을 주입 → 레시피별 프롬프트가 약해도 모든 생성에 근거 규율이 항상 적용된다.
  (코드 변경이므로 `run.bat` 재시작 후 반영.)
- **옵션 B(RAG) = 월요일 추가 의사결정.** 아래 5장 참고.
- **옵션 C(웹검색)·D(딥리서치) = 보류.** 인터넷 필수라 KOICA 보안망에서 어려움.

## 5. 옵션 B(RAG) 도입 시 무엇이 벌어지나 (월요일 의사결정용)

**한 줄 요약**: 생성 1건당 비용은 **가볍다**(검색 수십~수백 ms, 프롬프트에 약 1~2천 토큰 추가).
진짜 비용은 **초기 셋업(ChromaDB + 임베딩 모델)과 색인 자료의 품질·관리**다.
agy/codex는 구독 할당량 기반이라 토큰당 달러 과금은 없지만, 프롬프트가 커지면 할당량·속도·컨텍스트 한계에 영향이 있다.

### (1) 한 번 해둬야 하는 셋업
- **ChromaDB(벡터DB)** 구동 — 별도 프로세스(기본 포트 8100). 로컬에서 가능.
- **임베딩 모델** — 둘 중 하나:
  - 로컬 **fastembed**(ONNX, all-MiniLM-L6-v2 약 50MB, CPU, 서버 불필요, 최초 1회 다운로드) ← **오프라인·간편, 권장**
  - 또는 Ollama 등 임베딩 API 서버(별도 구동, `EMBEDDING_URL`).
- **색인(indexing)** — KOICA 기본자료(가이드·정책·과거 산출물)를 벡터DB에 한 번 넣는다(자료가 바뀌면 갱신).

### (2) 무겁나? (런타임)
- 생성 1건당: 질의 임베딩 + 벡터검색 ≈ 수십~수백 ms. **가볍다.**
- 무거운 쪽은 **셋업/색인(일회성)**. 수백 문서 색인 = CPU로 수분 수준.
- 메모리: ChromaDB + 임베딩 모델 상주. fastembed 모델은 작고, 청크 수천 개면 부담 적음.

### (3) 토큰 과다? (직답)
- RAG는 검색된 발췌(예: 상위 3~5개 × 약 500자)를 **프롬프트에 추가** → 호출당 입력 토큰 약 **+1,000~2,000**.
- agy/codex는 **구독(ChatGPT/Gemini) 할당량 기반** → 토큰당 직접 과금(달러)은 없다. 다만:
  - 프롬프트가 커져 **컨텍스트 한계**에 더 가까워지고, **속도 약간 저하**, **할당량/레이트리밋 소모 증가**.
  - 대응: top-k를 3 정도로, 발췌 길이 제한, "관련 있을 때만" 검색.

### (4) 오프라인 가능? — 예
- fastembed(로컬) + ChromaDB(로컬)면 **인터넷 없이** 동작 → KOICA 보안망에 적합(C·D와의 결정적 차이).

### (5) 연결 작업량 — 중간
- `_run_create`/`_build_prompt`에 검색 호출 추가 + 레시피 플래그(`rag_enabled`) + **RAG 다운 시 자동 우회**(graceful degrade).
- 별도로 **색인 도구**(자료를 벡터DB에 넣는 스크립트/UI)가 필요.

### (6) 위험·주의
- **쓰레기 입력 = 쓰레기 출력**: 색인 자료가 틀리거나 낡으면 오히려 잘못된 근거를 주입한다 → 색인 자료의 품질·최신성 관리가 핵심.
- 인프라 1개(ChromaDB) 추가 → 장애 지점 증가 → 반드시 graceful 우회 필요.
- RAG가 환각을 0으로 만들지는 못한다. **사람 검토는 계속 필요.**

### 월요일에 정하면 되는 것 (3가지)
1. ChromaDB + 임베딩(fastembed 권장) 셋업을 감수할지.
2. 어떤 KOICA 자료를 색인할지 → **아래 분석 참고**.
3. 정책값: top-k(예: 3), 발췌 길이, 어떤 레시피에 RAG를 켤지.
→ 이 3가지만 정하면 연결 구현은 가능하다.

### 색인 자료 후보 — AIM 엑셀 입력열 분석 결과 (근거 기반)
16개 마스터 산출물의 입력 열(C~J, 총 64개 참조파일)을 분석하니 두 부류로 나뉜다.

**(가) RAG에 색인할 것 — 모든 산출물 공통 "기관 기준자료"(안정적·반복)**
- **M&E guideline** (KOICA-NIGERIA model smart school M&E guideline) — 3개 산출물에서 반복.
- **PDM / 성과관리양식 / 지표정의서** (성과현황점검양식, 국영문 PDM) — 3개에서 반복. 용어·지표 일관성의 핵심.
- **표준 양식/Form** (Term Progress Report form, TLC Operation Plan form 등) — 4회.
- **사업수행계획서(RD)** — 사업 마스터플랜(있으면 1순위).
- → 이 4종을 색인하면 128개 산출물 전체가 올바른 지표 정의·용어·형식을 자동 참조(환각↓, 일관성↑).

**(가)+ 가장 큰 자산 — 1차 사업 전체 아카이브를 RAG 코퍼스로**
- 발주처(우리)가 **1차 사업을 직접 수행**했으므로, 그때 만든 계획서·결과보고서·회의록·가이드·PDM·용어 전체가
  "이 사업의 사실·표현 기준"이다. 이 더미 전체를 벡터DB에 색인해두면 128개 산출물 생성 시 **자동으로 1차 선례를 근거로** 끌어온다.
- 손으로 매번 올릴 수 없고 무엇이 필요할지도 미리 모르므로, **바로 이 경우가 RAG의 본령**이다.

**(나) RAG와 함께 쓰는 "직접 입력" — 단일 정답 선례를 콕 집을 때**
- 과거 동종 차수 산출물(예: 10차 계획서 ← 9차 계획서 1건)처럼 **정답 선례를 정확히 아는** 경우엔 그 1건을 입력으로 올리면 100% 반영(정밀).
- 회의록·참석자 명단 등 특정 이벤트 종속 자료도 그 건의 입력으로.
- 즉 RAG(넓게 깔린 기관기억) + 직접입력(콕 집은 선례)은 **배타가 아니라 함께** 쓴다.

> 규모 메모: 질의당 비용은 top-k(3~5)만 주입하므로 **코퍼스가 커져도 런타임은 가볍다**(검색이 추려줌).
> 무거운 건 일회성 색인뿐. 대용량 통합본(77MB·150MB 등)·중복·desktop.ini는 색인에서 정리/제외.
> 1차 아카이브 총량(문서 수·GB)을 알려주면 색인 시간·저장량을 가늠할 수 있다.
