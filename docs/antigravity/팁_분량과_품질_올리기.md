# 팁 — 문서 분량·품질 올리기 (들쑥날쑥 줄이기)

같은 레시피라도 **모델 + 프롬프트 강도**에 따라 결과 분량/품질이 크게 달라진다.
(예: 같은 입력에 Gemini 3.1 Pro ≈ 1장, Claude Sonnet 4.6 ≈ 4장, Opus ≈ 더 많고 표 위주.)
일관되고 충실한 결과를 얻는 방법:

## 1. 추가 지시/메모로 분량·구성 강제 (가장 쉬움)
생성 화면 "③ 추가 지시/메모"에 구체적으로:
> "다음 8개 섹션으로 작성: 1.개요 2.목적 3.인력(표) 4.주요과업(4.1~4.4, 각 절 표+세부활동)
> 5.일정표(주차별 표) 6.예산(안) 7.기대효과 8.기타. 각 절을 충분히 서술하고 A4 3~4장 분량으로."

→ 모델이 약하게 답하던 것도 구조·분량이 잡힌다.

## 2. 레시피 프롬프트에 구조/분량 박기 (반복 작업이면 권장)
`knowledge/recipes/*.yaml` 의 프롬프트에 위 "8개 섹션 + 분량" 지시를 넣어두면,
매번 메모 안 써도 항상 그 형식으로 나온다. (모델 바꿔도 동일 형식 → 모델 비교가 공정)

## 3. 샘플(예시) 문서 제공
잘 나온 예전 결과물을 "이 분량·목차·문체를 따르라"는 참고로 주면 그 수준으로 맞춰진다.
- 좋은 기준 샘플 예: `_assests/01-PMC-10/outputs/PMC-10차-파견계획서_*.md` (8섹션·표 다수의 충실한 완성본)
- 방법: 메모에 핵심 구조를 옮겨 적거나, 레시피에 few-shot 으로 포함.

## 4. 모델·추론강도 선택 ([⚙ 연결 상태])
- 더 길고 꼼꼼 → **(High)** 또는 **Thinking** 계열(Pro High, Sonnet/Opus Thinking).
- 빠르고 간결 → Flash (Low/Medium).
- 단, (High)/Thinking 은 **할당량·비용을 더** 쓴다([비용·할당량 비교](비용_할당량_비교.md)).

## 5. 모델 성향(관측 메모 — 채워가며 비교)
| 모델 | 분량 | 특징(관측) |
|---|---|---|
| Gemini 3.5 Flash | 짧음 | 간결, 빠름 |
| Gemini 3.1 Pro | 변동(약 1장 관측) | 짧게 나올 때가 있음 → 분량 지시 강화 필요 |
| Claude Sonnet 4.6 | 김(약 4장 관측) | 서술 풍부 |
| Claude Opus 4.6 | 더 김 | 양 많고 **표 위주**로 구조화 |
| GPT-OSS 120B | (테스트 중) | (기록 예정) |

## 핵심
**분량이 적게 나오면 모델 탓만 하지 말고 ① 메모로 구조·분량 지시 → ② 레시피에 고정 → ③ 샘플 제공**
순서로 강화. 그래야 모델 간 **품질 차이**가 공정하게 드러난다.
